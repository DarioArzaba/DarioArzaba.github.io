Medidas reales en CSS

Hace más de dos decadas las Macintosh-II contaban con una pantalla de 9 pulgadas y una resolución de 512x384. La resolución neta era de 72ppi con lo que el sistema operativo llego a utilizar la medida para inferir otras medidas reales (Pulgadas, Centímetros, Puntos).

Tiempo después las Mac obtendrán mejores pantallas con resoluciones más altas, sin embargo el OS continuó por mucho tiempo utilizando la resolución de 72ppi como referencia. En Windows pasaría algo similar, el OS continuaría utilizando su anticuada resolucion de 80ppi para calcular medidas reales.

Para los desarrolladores de sitios web, esto significaba que distintos dispositivos con distintas pantallas y distintos sistemas operativos interpretan una medida real de distinta forma. A pesar de que los OS actuales interpretan correctamente las medidas reales, ha quedado un estigma en su uso con relación al desarrollo de sitios web.

Content Sink: Objeto que se encuentra entre el parser y el DOM. Este objeto especial es el encargado de tomar los tokens que genero el parser e ir construyendo el DOM. En un servidor XML se consideran constructores SAX-DOM.

Cuando un documento XML (SVG, MathML, XHTML), es cargado en la memoria del navegador queremos que se construye incrementalmente en el caso de que el tiempo de renderizado del DOM tome demasiado. Actualmente este es el caso con HTML5 pero no con todas las implementaciones de XML.

Otro caso es cuando el documento se refiere e una transformación XSLT, en este caso el objeto (Content Sink) no debe de entregar el código al Layout Engine, lo que debe de hacer es construir el DOM sin el Layout y pasarlo intacto al XSLT Engine, ya será él quien se encarga de enviarlo al Layout.

En caso de que el documento no tenga CSS, XSLT o una interfaz para el DOM, entonces el objeto debe eliminar el todo XML Pretty Printer.

El Servidor Ideal

Era una tarde en la que mi servidor privado corriendo, uno de los discos duros falla, tengo que realizar un apagado forzoso con Cmd-Ctrl-Power. Después del reinicio la computadora se niega a encender (Ni idea si esta congelado o corriendo FSCK). Reinicio desde el DVD pero la utilidad no puede reparar la partición de Boot. Utilizo el snapshot de mi TimeMachine y sorpresa… esta corrupto. Se que ejecutar FSCK desde el DVD tomara todo el dia, opto por descargar mi Backup OffSite y reiniciar desde ahí.

Esta clase de cuestiones no debería suceder, necesitamos un NAS que exponga la info en NFS y utilice RAID para la redundancia. Que el sistema de archivos sea ZFS (Snapshots automáticos) y que puedan ser utilizados en NFS. Tambien seria ideal encriptar fácilmente con un USB y que el NAS realice un Obnam - Incremental - GPG-Backup de forma local para lugo subirlo a un SFTP remoto. Finalmente que todo esté integrado a una UPS inteligente y que nos permita cambiar fácilmente de cuenta de trabajo (NAS Local) a una cuenta de viaje (Sparse Image).

Ese seria el servidor ideal.
